    1. Module docstring explaining token budget purpose

    3. `from typing import Any`

    **4. `_get_tokenizer(model_name: str = "gpt-4o-mini")`** — internal helper per D-18:
    - `import tiktoken` (lazy import, only called when functions are used)
    - `return tiktoken.encoding_for_model(model_name)`

    **5. `_serialize_item(item: dict[str, Any]) -> str`:**
    - Serializes a single entity/relation dict for token counting
    - Format: `"\n".join(f"{k}: {v}" for k, v in item.items() if v is not None)`
    - Matches upstream LightRAG's typical serialization pattern for context assembly
    - Skips None values to avoid "key: None" tokens

    **6. `truncate_entities_by_tokens(entities: list[dict[str, Any]], max_tokens: int, model: str = "gpt-4o-mini") -> list[dict[str, Any]]`** (D-17):
    - Returns a prefix of the entities list whose cumulative serialized token count <= max_tokens
    - Algorithm: iterate entities, serialize each, accumulate tokens via `enc.encode(serialized)`, stop when count > max_tokens, return slice up to (but not including) the item that would exceed
    - If max_tokens <= 0, returns empty list
    - If first entity already exceeds max_tokens, returns empty list (no partial entities)

    **7. `truncate_relations_by_tokens(relations: list[dict[str, Any]], max_tokens: int, model: str = "gpt-4o-mini") -> list[dict[str, Any]]`** (D-17):
    - Identical algorithm to truncate_entities_by_tokens but operating on relation dicts
    - Uses same _serialize_item function
    - Separate function for caller clarity (entity vs relation context)

    **8. `compute_chunk_token_budget(total_tokens: int, sys_prompt_tokens: int, query_tokens: int, entity_tokens_used: int, relation_tokens_used: int, buffer_tokens: int = 200) -> int`** (D-17):
    - Calculation: `kg_tokens = entity_tokens_used + relation_tokens_used`
    - `remaining = total_tokens - (sys_prompt_tokens + query_tokens + kg_tokens + buffer_tokens)`
    - Returns `max(0, remaining)` — never returns negative
    - Follows upstream LightRAG calculation order exactly (RESEARCH.md Architecture Patterns: Pattern 4)
    - Per upstream pattern: `total_tokens - sys_prompt - query - entity_tokens_used - relation_tokens_used - buffer`
    - Default buffer of 200 tokens for prompt formatting overhead (RESEARCH.md Anti-Patterns)

    **9. Async wrappers** (D-19: thin async wrappers for Phase 4/6 pipeline):
    - `async def atruncate_entities_by_tokens(*args, **kwargs) -> list[dict[str, Any]]` — delegates to sync version
    - `async def atruncate_relations_by_tokens(*args, **kwargs) -> list[dict[str, Any]]` — delegates to sync version
    - `async def acompute_chunk_token_budget(*args, **kwargs) -> int` — delegates to sync version
    - Per D-19, core computation is pure (no I/O), async wrappers exist only for pipeline compatibility
