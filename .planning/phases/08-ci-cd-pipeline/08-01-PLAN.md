Workflow specification:
- **Name:** `Publish to PyPI`
- **Trigger:** `on.push.tags: ['v*']` (D-04)
- **Job name:** `publish`
- **Runner:** `ubuntu-latest` (matching deploy-mkdocs.yml pattern)
- **Steps:**
  1. `actions/checkout@v4`
  2. `actions/setup-python@v5` with `python-version: '3.12'`
  3. `actions/cache@v4` with `path: ~/.cache/pip`, key `pip-${{ runner.os }}-${{ hashFiles('pyproject.toml') }}`, restore-keys `pip-${{ runner.os }}-`
  4. **Set version from tag** (Bash): extract version from `${{ github.ref }}` by stripping `refs/tags/v` prefix, then `sed -i 's/^version = "[^"]*"/version = "'"${VERSION}"'"/' pyproject.toml`. Echo the extracted version for log visibility.
  5. **Build package** (Bash): `pip install build` then `python -m build`
  6. **Publish to PyPI** using `pypa/gh-action-pypi-publish@release/v1` with `password: ${{ secrets.PYPI_API_TOKEN }}` (D-01)

Do NOT include `id-token: write` permission (API Token auth, not OIDC Trusted Publisher per out-of-scope).
Do NOT use `pnpm/poetry/uv` — the build step uses `pip install build && python -m build` (standard Python build tool with hatchling backend).
Do NOT add a separate `build` job — single job is sufficient for this package.

The workflow file must match the indentation and YAML style of the existing `deploy-mkdocs.yml`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local repo → GitHub | Code including workflow pushed to GitHub; workflow becomes trusted execution context |
| GitHub Actions → PyPI | API token crosses from GitHub Secrets to PyPI publish action |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Tampering | publish.yml after push | accept | Git history preserves all changes; branch protection not in v2.0 scope (see out-of-scope: PR gate). Any malicious workflow edit is visible in commit log. |
| T-08-02 | Information Disclosure | PYPI_API_TOKEN in workflow logs | mitigate | GitHub Actions automatically redacts `${{ secrets.* }}` values from logs. The `pypa/gh-action-pypi-publish` action passes the token directly without echoing. |
| T-08-03 | Supply Chain | pypa/gh-action-pypi-publish action | accept | `pypa/gh-action-pypi-publish@release/v1` is the official Python Packaging Authority action — the most widely used and trusted PyPI publish action in the ecosystem. Pinning to `release/v1` provides stability while receiving security patches. |
| T-08-04 | Elevation of Privilege | Workflow write access | accept | The publish job has default GITHUB_TOKEN with no elevated permissions. No `id-token: write` or `contents: write` — the workflow only reads source and pushes to PyPI. |
| T-08-SC | Tampering | pip install in CI | accept | `pip install build` fetches from PyPI. The `build` package is a PyPA standard tool. Supply chain risk is inherent to any pip install but low for official PyPA tools. No npm/pip audit table required (Python ecosystem, well-known packages). |
</threat_model>
