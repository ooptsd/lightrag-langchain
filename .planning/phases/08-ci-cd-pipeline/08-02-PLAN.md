**Step 1 — Determine GitHub username:**
Run `gh api user --jq .login` to get the authenticated GitHub username. If `gh` is not available, ask the user for their GitHub username.
Store as variable: `GITHUB_USER`.

**Step 2 — Configure git remote:**
```bash
git remote add origin "https://github.com/${GITHUB_USER}/lightrag-langchain.git"
```

If a remote named `origin` already exists and points elsewhere, use `git remote set-url origin` instead.

**Step 3 — Update mkdocs.yml repo_url:**
Replace the `USER` placeholder in `mkdocs.yml` line 3:
```
repo_url: https://github.com/${GITHUB_USER}/lightrag-langchain
```
Commit: `docs(mkdocs): update repo_url to actual GitHub repo`

**Step 4 — Verify no secrets are staged:**
Run `git status` and confirm `.env` is NOT in the tracked/staged files. If `.env` appears tracked, run `git rm --cached .env` to untrack it while keeping the local file.

**Step 5 — Push main branch:**
```bash
git push -u origin main
```
This pushes all commits including the publish workflow (Plan 08-01) and the mkdocs.yml update. The `deploy-mkdocs.yml` workflow triggers on push to main per D-05.

**Step 6 — Recreate v1.0 tag to include CI workflow:**
The current local `v1.0` tag points to an old commit that predates the publish workflow. For the tag-triggered publish to work, the tag must point to a commit that includes `.github/workflows/publish.yml`.

Check where the current `v1.0` tag is:
```bash
git tag -l v1.0 && git log --oneline -1 v1.0
```

Force-recreate the tag on the latest commit (which includes the Plan 08-01 workflow file):
```bash
git tag -f v1.0 HEAD
```

Commit the tag update locally: `git push origin v1.0 --force`

**Important:** The force-push of the tag is necessary because the CI workflow file must be present in the tagged commit for the workflow to trigger. Without this, pushing `v1.0` would trigger a workflow on a commit that lacks `publish.yml`, resulting in no workflow execution.

**Step 7 — Push all other tags:**
```bash
git push origin --tags
```
This pushes any other local tags (if any exist) while reusing the already-force-pushed `v1.0`.

**Step 8 — Verify the push:**
Run `gh repo view --web` to open the repo and confirm all files are visible, including `.github/workflows/publish.yml`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local machine → GitHub | Code pushed over HTTPS; credentials via gh CLI or git credential helper |
| GitHub → PyPI | API token crosses from GitHub Secrets to PyPI publish action |
| GitHub → GitHub Pages | GITHUB_TOKEN (auto-provisioned) used by mkdocs gh-deploy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05 | Information Disclosure | .env accidentally pushed to public repo | mitigate | `.gitignore` excludes `.env`. Grep gate in Task 2 verifies `.env` is not tracked before push. If detected, `git rm --cached` removes it from tracking. |
| T-08-06 | Information Disclosure | .env.example exposes config structure | accept | `.env.example` uses placeholder values (`your_password`, `sk-your-api-key`) — no real secrets. The file is intentionally public as documentation. |
| T-08-07 | Tampering | Force-push of v1.0 tag | accept | The tag recreation is necessary because the initial `v1.0` tag predates CI workflow creation. After this one-time force-push, subsequent releases use normal tag creation (`git tag v1.0.1 && git push origin v1.0.1`). Git history preserves the original tag location. |
| T-08-08 | Elevation of Privilege | deploy-mkdocs.yml has `contents: write` | accept | The `contents: write` permission is required for `mkdocs gh-deploy --force` to push to the `gh-pages` branch. This is the documented GitHub Pages deployment pattern and is scoped to the GITHUB_TOKEN which expires after the workflow run. |
| T-08-09 | Repudiation | No way to trace who triggered a release | accept | Git tags are immutable (after push) and associated with the pusher's GitHub identity. Workflow runs show who pushed the tag. For v2.0, this is sufficient — OIDC/signing is out of scope. |
</threat_model>
