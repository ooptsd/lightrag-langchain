<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User query → chain.ainvoke() | Untrusted user input enters the pipeline — passes through keyword extraction (LLM with user input in prompt), retriever query embedding, and LLM answer generation |
| system_prompt parameter → chain.ainvoke() | D-08 allows caller to replace entire system prompt — caller-controlled content becomes the LLM's instruction set |
| Retriever output (Document list) → token budget | Trusted internal data — Documents are produced by Phase 5 retriever from database. Metadata integrity assumed. |
| Prompt templates (.format()) → LLM | Template placeholders filled with potentially user-originated data (file_path from DB, query text) |
| LLM response → chain output dict | LLM output is opaque — chain does not parse or validate LLM response content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | chain/base.py — user query injected into keyword extraction prompt | accept | query text is embedded in a structured system prompt with explicit "only use context" instructions. The keyword extraction prompt template constrains output to JSON schema via `with_structured_output(KeywordsSchema)`. No custom sanitization — upstream template design is the mitigation. |
| T-06-02 | Tampering | chain/base.py — system_prompt parameter (D-08 full override) | accept | D-08 is a documented feature allowing full system prompt replacement. The caller is responsible for the content they inject. This is an intentional design decision for flexibility (e.g., different languages, custom instructions). |
| T-06-03 | Spoofing | chain/base.py — file_path in Document metadata used for source citations | accept | file_path originates from LightRAG ingestion pipeline (database-stored), not from user input. If an attacker can inject malicious file_path values into the database, the threat is at the data ingestion layer (out of scope). The chain layer displays file_path as-is in citations. |
| T-06-04 | Information Disclosure | chain/base.py — retrieved documents may contain sensitive content passed to LLM | accept | This is inherent to RAG systems — the retriever returns whatever is in the database. Content filtering at ingestion time (LightRAG pipeline) is the correct control point. The chain does not add or remove content from retrieved documents. |
| T-06-05 | Tampering | chain/base.py — template injection via `.format()` placeholders | mitigate | All prompt templates use named `.format()` placeholders (e.g., `{context_data}`, `{entities_str}`). User-supplied values (query, file_path) are passed as `.format()` keyword argument VALUES, not as part of the format string itself. Python's `str.format()` does not recursively expand placeholders in substituted values. f-strings are never used for user data. |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | slopcheck. No new packages in Phase 6 — all dependencies (pydantic, langchain-core, langchain-openai, tiktoken) already installed and verified by prior phases (RESEARCH.md Package Legitimacy Audit). |
