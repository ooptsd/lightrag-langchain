# Roadmap: lightrag-langchain

## Overview

A Langchain-based read-only query layer for LightRAG's pre-processed PostgreSQL knowledge graph. The project builds from configuration through data access, LLM integration, query strategy implementation, Langchain retriever wrapping, and finally a complete end-to-end QA chain. Each phase is a complete technical layer that the next phase builds upon.

## Phases

- [ ] **Phase 1: Configuration** - .env-based configuration for all services (PG, LLM, Embedding, Reranker, QueryParams)
- [ ] **Phase 2: Data Layer** - Read-only PostgreSQL access layer for LightRAG's PGVector and Apache AGE graph stores
- [ ] **Phase 3: LLM Integration** - ChatOpenAI, OpenAIEmbeddings, multi-reranker, keyword extraction, and token budget control
- [ ] **Phase 4: Query Strategies** - All 6 LightRAG query modes: naive, local, global, hybrid, mix, bypass
- [ ] **Phase 5: Retriever Interfaces** - Langchain BaseRetriever subclasses wrapping each query mode with sync/async support
- [ ] **Phase 6: QA Chain** - Full LCEL Chain: query to retrieval to context assembly to LLM generation

## Phase Details

### Phase 1: Configuration
**Goal**: All configuration is loaded from .env and accessible to every downstream layer via a typed configuration API.
**Depends on**: Nothing (first phase)
**Requirements**: CONF-01, CONF-02, CONF-03, CONF-04, CONF-05
**Success Criteria** (what must be TRUE):
  1. Project module can be imported without errors (package structure, dependencies, init files in place)
  2. `.env` file is parsed and all config values are accessible through a configuration API with typed fields
  3. Missing required configuration raises a clear, actionable validation error indicating which key is missing
  4. Each of the 5 config groups (PostgreSQL, LLM, Embedding, Reranker, QueryParams) is independently loadable and testable in isolation
**Plans**: TBD

### Phase 2: Data Layer
**Goal**: All LightRAG PostgreSQL data (entities, relationships, chunks, graph nodes/edges) is readable through a clean abstraction layer.
**Depends on**: Phase 1
**Requirements**: STOR-01, STOR-02, STOR-03, STOR-04
**Success Criteria** (what must be TRUE):
  1. PGVector `entities_vdb` table can be queried for vector similarity search returning entity records with name, content, source_id, and file_path
  2. PGVector `relationships_vdb` table can be queried for vector similarity search returning relationship records with src_id, tgt_id, content, keywords, and weight
  3. PGVector `chunks_vdb` table can be queried for vector similarity search returning chunk records with content, full_doc_id, chunk_order_index, and file_path
  4. Apache AGE graph can be traversed: entity nodes by ID with their properties (entity_type, description, source_id) and relation edges between them with their properties (description, keywords, weight)
  5. All database operations are confirmed read-only -- no CREATE, INSERT, UPDATE, or DELETE statements are generated by the data layer
**Plans**: TBD

### Phase 3: LLM Integration
**Goal**: LLM, embedding, and reranker services are integrated with provider-agnostic interfaces and token budget enforcement.
**Depends on**: Phase 1
**Requirements**: LLM-01, LLM-02, LLM-03, LLM-04, LLM-05
**Success Criteria** (what must be TRUE):
  1. Any OpenAI-compatible LLM provider can be used for text generation by setting `LLM_BINDING`, `LLM_BINDING_HOST`, and `LLM_BINDING_API_KEY` in .env
  2. Embeddings are generated via OpenAI-compatible API (configurable `EMBEDDING_BINDING`) for encoding query text into vectors matching the upstream 1024-dimension format
  3. Multiple reranker backends can be used interchangeably (aliyun gte-rerank-v2, cohere, jina) by changing `RERANK_BINDING` in .env, all producing scored result lists
  4. High-level keywords (macro themes) and low-level keywords (specific entities) are extracted from user queries via LLM with a structured output prompt
  5. Token budget is enforced: entity tokens + relation tokens do not exceed max_total_tokens, and remaining token capacity is dynamically allocated to chunk content

### Phase 4: Query Strategies
**Goal**: All 6 LightRAG query modes produce correct, distinct retrieval results matching the upstream LightRAG behavior.
**Depends on**: Phase 2, Phase 3
**Requirements**: QUERY-01, QUERY-02, QUERY-03, QUERY-04, QUERY-05, QUERY-06
**Success Criteria** (what must be TRUE):
  1. Naive mode queries chunks_vdb by vector similarity only (using configured COSINE_THRESHOLD and CHUNK_TOP_K), returns ranked chunk list with no graph traversal
  2. Local mode performs entities_vdb vector search for top-K entities, then traverses AGE graph to collect associated edges and neighbor entities, returns entity-centric results
  3. Global mode performs relationships_vdb vector search for top-K relations, then traverses AGE graph to retrieve associated entity nodes for each relation, returns relation-centric results
  4. Hybrid mode executes local and global retrieval in parallel, merges results via round-robin interleaving, and returns combined deduplicated results
  5. Mix mode executes hybrid retrieval and additionally searches chunks_vdb, merging graph knowledge with raw text chunks into a single ranked result set
  6. Bypass mode returns an empty retrieval result (no documents retrieved), delegating entirely to the LLM for answer generation
**Plans**: TBD

### Phase 5: Retriever Interfaces
**Goal**: Each of the 6 query modes is exposed as a standards-compliant Langchain BaseRetriever with full sync/async support and source attribution metadata.
**Depends on**: Phase 4
**Requirements**: RETR-01, RETR-02, RETR-03
**Success Criteria** (what must be TRUE):
  1. Each of the 6 query modes has a corresponding BaseRetriever subclass (NaiveRetriever, LocalRetriever, GlobalRetriever, HybridRetriever, MixRetriever, BypassRetriever) that can be instantiated and imported
  2. All retrievers support synchronous `invoke(query: str) -> List[Document]` and asynchronous `ainvoke(query: str) -> List[Document]`, returning Langchain Document objects with page_content and metadata
  3. Retrieved Document metadata consistently includes source_id, file_path, and mode-specific attribution (entity names for local/global, relation descriptions for global, chunk_order_index for naive/mix)
  4. Retrievers are composable with standard Langchain patterns -- they work correctly when used as standalone retrievers and when passed to Langchain chain constructors
**Plans**: TBD

### Phase 6: QA Chain
**Goal**: A complete end-to-end LCEL Chain that takes a user query and returns a generated answer with source citations, supporting all three invocation modes.
**Depends on**: Phase 5, Phase 3
**Requirements**: CHAIN-01, CHAIN-02, CHAIN-03
**Success Criteria** (what must be TRUE):
  1. Chain executes the full pipeline: user query enters, keywords are extracted via LLM, retriever fetches context documents, context is assembled within token budget, and LLM generates a final answer with inline source citations
  2. Chain supports synchronous `invoke(query: str) -> dict` returning answer text and source references
  3. Chain supports asynchronous `ainvoke(query: str) -> dict` for non-blocking execution and `astream(query: str) -> AsyncIterator[dict]` for streaming token-by-token output
  4. Pre-provided `hl_keywords` and `ll_keywords` in QueryParam bypass the LLM keyword extraction step, proceeding directly to retrieval with the given keywords
  5. Generated answers include traceable source citations referencing the original document file_path, source_id, or entity/relation identifiers from retrieved context
